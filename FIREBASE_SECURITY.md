# Firebase セキュリティ 確認結果と対処

## 現状の評価（2026-07-17 監査）

| 項目 | 状態 | リスク |
|---|---|---|
| Firebase apiKey | `index.html`/`sanma_combined.html` にハードコード（PUBLICリポジトリに公開） | **低**。Firebaseのウェブapikeyは仕様上クライアント公開前提でGoogleも「秘密ではない」と明記。ただしドメイン制限は推奨 |
| Realtime Database ルール | セットアップ手順が「**テストモードで開始**」を案内 → 実質**誰でも全読み書き可能**の可能性が高い | **中〜高**。DBのURLさえ分かれば第三者が全ルーム・全ユーザーを閲覧/改ざん/削除できうる |
| 認証方式 | Firebase Authを**使っていない**（独自ID＋SHA-256ハッシュを`users`配下に保存し、クライアントが読み取って照合） | **中**。パスワードハッシュがクライアントから読める設計。ルールだけでは根本解決不可 |

> ※本リポジトリはあなたの業務外（個人の三麻ゲーム）で、賭け等の実害はないため深刻度は限定的。ただし公開DBの放置は避けるべき。

## 対処1（すぐできる・非破壊）: セキュリティルールの適用
テストモードの全開放を、同梱の [`database.rules.json`](database.rules.json) に置き換える。
- ルート直下の一括読取・一括削除を**拒否**（DB丸ごとの吸い出し・全消去を防止）
- 既知パス（`users` / `rooms` / `room_*`）のみに読み書きを限定
- キー長のバリデーションでストレージ濫用を抑制

**適用手順**
1. Firebase Console → 対象プロジェクト → 構築 → Realtime Database → 「ルール」タブ
2. `database.rules.json` の内容を貼り付け → 「公開」
3. ゲームを一通り動かし、ルーム作成・対戦・戦績保存が正常か確認

## 対処2（推奨・非破壊）: APIキーのドメイン制限
第三者による無断利用を抑える。
1. Google Cloud Console → APIとサービス → 認証情報 → 対象のAPIキー
2. 「アプリケーションの制限」→ **HTTPリファラー** → 配信ドメインを追加
   （例：GitHub Pagesなら `https://<ユーザー名>.github.io/*`）
3. 「APIの制限」→ Firebase利用に必要なAPIのみに限定

## 対処3（根本対策・要改修）: 匿名認証の導入
パスワードハッシュをクライアントに読ませる現設計を、Firebase Authに置き換える。
- Firebase Console → Authentication → Sign-in method → **匿名（Anonymous）** を有効化
- アプリ起動時に `firebase.auth().signInAnonymously()` を実行
- ルールを `".read": "auth != null"` / `".write": "auth != null"` に強化
- ユーザー認証は Firebase Auth（メール/パスワード or カスタムトークン）へ移行し、
  `users`配下からパスワードハッシュを排除する

→ ここまでやると「DB URLを知る第三者による読み書き」を実質遮断できる。
   個人ゲームで工数を掛けたくない場合は、対処1＋2だけでも現状よりは大幅に安全。
